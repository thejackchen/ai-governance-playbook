# MIGRATION.md — 存量项目的绞杀式迁移(六步)

> 读者:安装 AI。你在 [setup.md](setup.md) 第 0 步 Q2 判定项目是存量,被引到这里。读完本文件回 setup 第 1 步照常走——本文件只改变其中几步的动作:**从「新建」改为「映射既有资产」**。
> 动手前先通读本仓库 [governance/cases/](governance/cases/README.md) 的现有判例——前人踩过的坑不再踩,迁移中遇到同族场景直接类比引用。
> 铁律:**绞杀式 = 保留旧文件加降级注,不删除、不搬家、不改写历史;回滚就是删掉新文件、摘掉降级注,一切如旧。** 大爆炸重造是本模板明确反对的路线。

## 第 1 步:盘点现状规则(只看,不动手)

把项目里一切「管 AI/协作行为」的东西列成一张清单:旧宪法或行为守则、README 里的约定段、散落的规范文档、既有 hook 与 CI 检查、乃至口头惯例(问负责人一句)。每条记:内容一句话 + 现在写在哪 + 实际有没有人/机器在执行。
产出是一张临时清单(任务产物,不必入库)——它是后面五步的原料。

## 第 2 步:registry 上户口 = 给旧规则记执行率账

对照 [templates/governance/registry.md](templates/governance/registry.md) 的六字段,把清单里的规则逐条上户口。这一步是**记账,不是审判**:

- **载体列如实填**:纯文档约定就是 L3,哪怕它措辞再严厉——这正是本模板的核心洞察(执行率由介质决定,与措辞无关);
- **出处列答不出的**(哪次真实失败证明它值得留)标「复审候选」,**先不删**——删除交给心跳季审,用「零触发」证据说话,别在迁移日凭感觉大清洗;
- 说不清解决哪个结构问题(S1–S5)的,挂 `[cap]`(执行者能力补丁,换代清仓);
- **≤30 条上限在这一步就生效**:装不下就是信号——优先收编有出处的真规则,其余降级 L3 参考,不进台账。

## 第 3 步:按九段重写宪法(旧治理文件降级加头注,不删)

以 [templates/CLAUDE.md](templates/CLAUDE.md) 为骨架,从现状清单中把值得每 session 在场的内容收进九段(意图纲领仍必须经负责人确认)。然后给每份旧治理文件**头部加一行降级注**:

> 本文件已降级为 L3 深度参考({日期});行为层权威见 /{宪法文件名}。

旧文件原地保留。这一行就是绞杀式的全部成本,也是回滚开关:新宪法不好用,删新文件、摘降级注,零损失回到旧世界。

## 第 4 步:五件套衔接既有台账(历史不迁移)

- 已有踩坑记录/事故清单 → 新 `governance/incidents.md` 头部放一行归档链接(「历史见 X,新条目自此记录于本文件」),**历史条目原地留档,不搬**;
- 已有 ADR/决策目录 → 宪法与 docs/index.md 直接指向它,不新建、不搬家;
- 已有 FAQ/裁决记录 → 其中体现负责人品味的,值得按六字段改写成前几个判例(这是唯一建议「搬」的内容,因为判例是增值资产);其余留原地。

历史不迁移的理由:迁移历史 = 大爆炸行为 + 改写只追加的账本,两条都踩线。

## 第 5 步:心跳安装(与全新项目无差别)

按 setup 第 1.5 步照做。存量项目的第一期对账包格外有用:它会立刻暴露第 2 步里那些「上了户口但从没人执行」的规则(活性读数、队列空转、复审候选堆积)——让证据替你完成第 2 步没做的裁汰。

## 第 6 步:新旧对照验证(建议做一次)

挑 1–3 个近期真实任务,让一个**只读新治理**的全新 session 重做或评审,与旧治理下的产出对照(有条件就匿名盲判)。至少验证最低线:新 session 只靠宪法 + ROADMAP,能否正确说出「项目在干什么 / 此刻在哪 / 哪些不能碰」。
对照结果记入 CHANGELOG;发现新治理缺关键信息 → 当场补进对应文件。这就是 setup 第 2 步「首任务演练」的存量加强版,做完直接进 setup 第 3 步自检交付。

## 新旧并存期的三条保证

1. 旧文件一个不删(降级注可摘除 = 完全可回滚);
2. 历史台账一条不搬(append-only 不因迁移破例);
3. 并存期行为权威唯一:降级注写明白「以新宪法为准」——两个权威并存比没有权威更糟。
