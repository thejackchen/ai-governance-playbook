# 治理架构精华（可迁移版 · 白纸重构后）

> 这是一个 **AI 主导维护**的软件项目治理体系的提炼。写给另一个 AI（或建治理的人）看——读完你应能借鉴**为什么这么设计**（思想 / 原理 / 踩坑），而不是抄具体实现（那是项目特定的，换个项目未必合用）。
> 全文重「思想」，每条尽量带「踩坑 why」——**踩坑教训是最值钱的部分**，标 ⚠️。
> **这是第二版。** 第一版推荐过一套重型架构（三线一底座 / 控制点注册表 / 季审 / L1–L4 真理）。母版项目用了 6 周把它跑成了官僚，然后**白纸重构**砍掉了重的部分——本文记录新架构 + 那个「知道病却开重药、再自我重构」的诚实故事（§7）。第一版里**清醒的定律 / 反模式 / 最诚实那条原封保留**（§8、§9），被砍的只是它开的「重药」。
> **定位：本文是「给人理解为什么」的完整方法论。** AI 建治理只读一次性安装指令（`setup.md`），本文讲那些命令背后的原理、机制、踩坑。

---

## 1. 总纲：把治理当「成本」精算，别当「资产」积累

业内所有验证过的最佳实践（Anthropic / OpenAI 官方 agent 指南、karpathy、ADR、OpenSpec、fitness functions、policy-as-code）有一条共性：**它们把规则 / 文档 / 流程当「成本」精算**——消耗最稀缺的资源（人脑注意力、AI 的 context 预算），**不当「资产」积累**。

⚠️ **母版项目的病根就是反了这条**：把治理当资产（越全越放心、写了就留），于是只增不减。6 周长出 9 个「关于治理的治理」决策、20 个 lint 检查（只 3 个真拦得住）、数万行文本，却**从没证明过自己帮产品防了一次事故**。

总纲一句话：**任何治理零件默认是负债，要由真实发生的失败证明自己值得留下。** 这是后面四条原理、所有「砍」动作的根。

## 2. 四条不变原理（治理自身的「不变层」）

> 规则会烂、原理不烂。把「不变的」放进原理，「会变的」交给按原理代谢的机制。这是「人管不变、AI 管会变」的延伸——过去「不变的」只写了使命，**这四条把「治理自身的不变原理」也写进了北极星**。任何治理动作过这四问，过不了不做。

### 原理一 · 为消费者而生，无消费者即废

一切规则从「**谁用、在哪用**」倒推：交接为接手的 AI 写、ADR 为未来的读者写、状态文件为下一个无记忆 session 写。**没有消费者的规则不是中性的，是负担。**

- **why / 踩坑** ⚠️：一个有维护成本的标记 / 指标，若没有任何决策或行为真的读它、因它改变行为，它**必然衰减成装饰**。母版实例：一套「成熟度三级制」洞察标签，几十个洞察只有个位数有验证记录——因为 AI 引用时从不按等级改行为。装饰化可机械检测：「最近 30 天被谁读过、影响过什么决策」答不出 = 删除候选。
- **它印证 / 砍了旧版哪条**：✅ 完整保留旧版第一定律「信号要有消费者」（§8）和「出口校验：治理的消费者必须是产品，不是治理自己」。这条没被砍——它是这次重构的**判据本身**。

### 原理二 · 腐烂是默认，解药是「持续自动」而非「周期意志」

文档 / 规则 / 结构默认会腐烂。唯一扛得住的是把校验**绑进每次工作流、自动跑**（每轮 hook / commit / CI），不是靠人定期来一次维护仪式。

- **why / 踩坑** ⚠️：实测对照——有门禁（lint / hook）的机制全部存活；纯靠自觉的维护仪式全部衰减（断档以周计）。fitness functions（《演进式架构》）给了行业学名：架构默认腐烂，除非持续自动验证；书中**点名要淘汰的失败模式正是「周期性人工评审」**。
- **它砍了旧版哪条** ❌：旧版第二定律「有门禁存活、纯自觉衰减」✅ 保留（§8.2），但旧版给它配的解药是「**季审 cron**」——母版 6 周里**那个季审一次都没跑过**。死触发器上挂着所有自我纠错（ROI 举证 / 仪式预算 / 自觉桶清理），链条最后一环断了，前面只增不纠。**新架构砍掉季审，换成事件驱动纠错**（§5）：纠错绑到「lint 报错 / evals 答不出 / 用户打脸 / 撞见过时规则」这些本来就高频自动发生的事件上。同一条定律，旧版自己开了一副「会衰减的周期药」，违背了自己。

### 原理三 · 增量生长，沉淀进唯一真相（SoT）

只记「**在变什么**」（delta），完成后归档进一个生长的唯一真相源；不一次写大而全，不多处抄副本。

- **why / 踩坑** ⚠️：真相散落多处 + 一个模糊数字被当成全量 → 动手前靠记忆拼凑 → 拼错 → 重复劳动 → 结论又不回写权威处 → 散落依旧。**这是个恶性循环。** 业内对应：OpenSpec 的「delta + archive into growing SoT」、ADR 决策日志、单一信源 + 其余指针。
- **它印证了旧版哪条**：✅ 保留旧版「散落导致重复」定律（§8.4）和「声明式解析优于缓存状态」（§8.9：读时解析没有「赢家状态」，数据变下次读自动反映、零同步）。新增的是把它升成正式原理——**SoT 是唯一的根**：做产品的方法 = 治理自己的方法（§3）。

### 原理四 · 最简默认，复杂度要「挣来」

每条规则**默认不存在**，必须由一次真实发生的失败证明自己值得留。按真实失败迭代，不按想象的风险预防。

- **why / 踩坑** ⚠️：DHH/37signals「别因一次划伤就留疤」——别出一次错就加一条永久规则，正中「踩坑→加检查→越治越重」的增生病。另一面（⚠️ 这次重构自我修正过）：**「短」不是为人脑记忆，是为 AI 的 context 预算**——实测 LLM 只能可靠遵循约 150–200 条指令、配置 >500 行多被忽略、超字节阈值静默截断。所以治理文件该短，理由从「怕人记不住」改成「保住 AI 的高信号 token 预算」（§4）。
- **它砍了旧版哪条** ❌：旧版「治理重认知不重形式」（§8.7）✅ 保留，但旧版仍保留了「行数硬上限当人脑负担约束」「仪式预算点数」「one-in-one-out」这类**为凑形式而生的机制**。新架构砍掉这些形式机制本身——保留「该短」，但靠原理判断而非硬数字仪式。

## 3. SoT 根：做产品的方法 = 治理自己的方法

北极星的第一句话不是抽象使命，是一条**根**：我们做的产品本身就是「把散在各处的事实收进一个唯一真相源（同一个事实只有一个权威出处，不许多份副本各说各话）」。

**这条根也管治理自己**——知识、规则同样一处权威、不抄副本。**做产品的方法，就是治理自己的方法。**

- **why / 踩坑** ⚠️：旧版宪法写的是「L1–L4 四条真理」（优先级 L1>L2>L3>L4）。母版复盘发现**连用户自己都看不懂 L1–L4**——北极星对它的第一消费者失效（违原理一）。重写成大白话的「SoT 一条根」后，无记忆 AI 一读就能复述。**北极星该用产品的语言、不用治理黑话**：抽象的优先级编号是「治理为治理生产」的味道。
- 北极星仍只装「不变的 + 人管的」：使命（SoT 根）+ 四条原理（§2，当判断标尺）+ 几条铁律 + 修订机制（AI 不得自动改，要改用专门 commit 前缀 + 用户拍板）。它该短**因为要常读**，不是因为怕记不住。

## 4. 为 AI 写（不是为人脑写）

真正天天读治理文件的是 AI——每次对话开头自动塞进它的 context。所以治理文件的设计目标是 AI 的认知，不是人的记忆负担。

四条来自 Anthropic / OpenAI 官方 context 工程指南的具体约束：

- **context 是稀缺资源，边际递减**：北极星指标 = **「最小的高信号 token 集」**（可量化体检，不是凭感觉）。
- **context rot** ⚠️：token 越多，模型回忆力越降。所以「宁全勿简」要看场景——**人管的入口文件求最小高信号**（短为性能）；**AI 自主编译、按需检索的知识库才宁全勿简**（不进开头 context、用时才取）。把「精简到人脑可记」套到知识库是坑（恰与 karpathy LLM-wiki 原意相反：人管的才精简，KB 内容宁全勿简）。
- **right altitude**（别太具体也别太抽象）：印证「北极星 / 主入口 / 操作手册」分三层高度是对的，不该全推平成一个大文件——太抽象落不了地，太具体淹没信号。
- **跨工具同源**：不同 AI 工具读不同入口文件名，但要指向**同一份规则**（一个文件 import 另一个，或双文件同内容），且各工具的每轮 hook 跑**同一个 lint 脚本**。⚠️ 否则规则在两个工具间漂移、铁律只对一个工具可见。

⚠️ **元教训**：母版的重构提案自己讲「为 AI 写、要可执行」，却被无记忆 AI 实测出 **45% 是「读着深刻、对 AI 落不了地」的愿景词**（「最小高信号」无度量、「自动新陈代谢」无机制、「归档成 story」无文件形态）。**口头保证不可信——这正是为什么要 §6 用 AI 验 AI。**

## 5. 事件驱动纠错（取代季审）

纠错不靠「定期审查」（季审是死触发器），而是绑到本来就会**自动 / 高频发生**的事件。事件一来，照表纠：

| 事件（自动 / 高频发生） | 纠错动作 |
|---|---|
| **lint 报 error / warn**（每轮结束、commit 时都跑） | 当场修到 0 error，别留到以后 |
| **evals 某题答不出**（大改 / 换模型时派无记忆 AI 跑） | 改那道题对应的**承载文件**（修文件不修答案）——答不出 = 那块治理失效了 |
| **用户质问 / 打脸**（"这不对"、"你又附和了"） | 当场改 + 沉淀（踩坑进知识库、协作偏好进上手包） |
| **干活时撞见规则过时 / 自相矛盾** | 当场改（SoT 一处权威只改一处）+ 记踩坑。**过时规则 = 有害，删不留**（错的指令比没指令更糟） |

每条都写清「谁触发、做什么」，触发器本来就在工作流里跑——所以**不需要单独的纠错仪式或季审**。这就是原理二「持续自动 > 周期意志」落到地上。

**加规则前过三问**（原理落地，过不了不加）：① 有具体消费者吗？② 是真实发生过的失败挣来的，还是预防想象出来的？③ 机器拦得住吗——拦不住的别塞进 lint，靠 evals + 用户在环。
**砍规则的判定**：没有任何文件链它（孤儿，grep 反链可判）+ 长期无触碰 → 标候选、确认后砍。

⚠️ **为什么这是真机制不是换皮**：旧版「事件驱动」一度也是口号——被无记忆 AI 一问「谁捕获事件、自动触发什么动作」就露馅（§4 的 45% 病）。真机制的判据是：**每个触发器都已经在工作流里自动跑**（lint 在每轮 hook + CI 上、evals 在换模型时、用户打脸天然高频），不依赖任何人「记得定期来一次」。

## 6. 用 AI 验 AI（验收套件 + 无记忆 AI 跑）

治理有没有用，不靠口头保证，靠**考试**：起一个**无本会话记忆的冷上下文 AI**（换工具、或清空记忆），只许读治理文件，给它真实任务，看它做不做得对。

- **两层**：① **evals 题库**（轻量、日常 / 换模型时跑，~10 题测「接手 + 理解 + 铁律」）② **验收套件**（治理大改后跑一遍完整端到端回归，覆盖全生命周期：接手 → 干活铁律 → 对话路由 → 知识沉淀 → 结束前 → 自纠）。
- **每例含机器可查判据**（grep / git diff / 工具轨迹）**+ 语义评分要点**——不是纯主观打分。
- **判据**：连续两题失败的领域 = 那块承载文件失效，**修文件不修答案**。
- **触发**：模型换代 / 主力工具更换 / 治理大改时跑（**不排期季审**）。

⚠️ **为什么不可省**：lint 验「文件结构对不对」（确定性），evals 验「AI 行为遵不遵从」（概率性）——这是架构里**唯一能测「模型换代鲁棒性」和「自觉级规则衰减」的仪器**。母版实测：重构提案被 fresh AI 打出「45% 落不了地」，全靠这把仪器抓出来；冷启动自足化也靠它验过 8/8。**口头说「为 AI 写、可执行」都不可信，跑一遍才知道。**

⚠️ **冷上下文是硬要求**：考试必须由无本会话记忆的 agent 跑，**不得当班自审**——带着上下文的 AI 会脑补出文件里其实没写的东西，测不出冷启动缺口。

## 7. 演化故事：知道病、却开了会得病的重药，然后自我重构

> **这是本文最值钱的一节。** 它是一个治理体系「自己的定律预言了自己的死法、自己却仍开了那副药、6 周后自我重构」的诚实记录。

**第一幕 · 立框架（旧架构）。** 项目治理从一份 163 行的项目文件 + 一堆散落决策起步，没有一个「不变的真理」文件。于是立了北极星（L1–L4 四条真理）+ 三条内容线（治理 / 状态 / 沉淀）+ 执行底座。这是对的方向。

**第二幕 · 加重药（病根埋下）。** 接着把它收成「三线一底座」，并加上：控制点注册表（任何机制出生先登记 4 字段）、元门禁（「守护守护者」：机器查有没有未登记的规则）、仪式预算（每轮仪式 ≤N 个）、season 季审（4 维 filter + ROI 举证 ≥3 例）。**这是自创框架的高峰——也是病根。** 越来越多机制管的是「治理自己」，不是产品。

**第三幕 · 定律预言了病，药却照开。** 讽刺的是，**这套架构自己的定律早就写下了诊断**：

- 它的**第一定律**写着「信号要有消费者，没人因它改行为的标记必衰减成装饰」——可它自己开的「成熟度三级标签」「仪式预算点数」恰恰没有消费者。
- 它的**出口校验**写着「治理的消费者必须是产品，不是治理自己」——可它新增的 9 个决策里多数是「关于治理的治理」。
- 它的**第二定律**写着「纯自觉的衰减」——可它把所有自我纠错（ROI 举证 / 仪式预算清理）全挂在「季审」这个**纯靠人记得去跑**的触发器上，6 周一次没跑过。

**它知道病。它的定律就是病的诊断书。但它仍开了一副会得这个病的重药。**

**第四幕 · 白纸重构。** 用户一句「所有架构都可以重构，给我你觉得好的方案」触发复盘。博采众长（OpenAI / Anthropic 官方 + karpathy + ADR / OpenSpec / fitness functions / policy-as-code），定了一条判据——**凡业内验证过的留，凡自创的「管治理的治理」砍**：

- 北极星 L1–L4 黑话 → 重写成大白话「SoT 一条根」（连用户都看得懂了）。
- 主入口从「80 行政策指针」→ 39 行「为 AI 写的真交接」。
- lint 从 20 检查 → 8。
- **砍掉**：控制点注册表 / 元门禁 / 季审 / 仪式预算 / ROI 举证制度 / 三线一底座。
- **保留**：知识沉淀复利、机器兜底、evals 测行为、karpathy 铁律、SoT——这些是业内印证过的内核。
- 重构每一步用 **evals + lint 当客观判据**，确保「改完更好不是更乱」（白纸重构后跑了 18/18 端到端回归）。

**这个故事的教训**（给下一个建治理的 AI）：

1. ⚠️ **「知道定律」≠「不犯定律」**——你完全可能一边写下「信号要有消费者」，一边亲手造没有消费者的信号。**定律是事后诊断的尺，不是事前免疫。** 所以要 §6 用 AI 验 AI，定期拿冷 AI 把自己的治理重新量一遍。
2. ⚠️ **自创「管治理的治理」是最隐蔽的负债**——它感觉很严谨（「守护守护者」多聪明），但它没有产品消费者，纯粹是治理在喂治理。**最聪明的元层机制，往往是最该砍的。** 判据：它防的是产品事故，还是治理自己的不规范？后者就是候选。
3. ⚠️ **重构本身也会落进同一个坑**——「重新发明一套治理框架」是最大的反模式（掉回造资产的循环）。母版的红线是：**只删和合并、不发明新框架**，且用 evals 当「改完没变乱」的客观判据。这次重构能成，是因为它克制住了「再造一套更好的」的冲动。

## 8. 保留的清醒定律 / 反模式（旧版原封不动，去项目特定后）

> 这些是第一版里最清醒的部分——每条都是真金白银的踩坑。新架构砍的是旧版的「重药」，**这些定律一条没动**。

**定律**：

1. **信号要有消费者** ⚠️：没人因它改行为的标记 / 指标必衰减成装饰。「最近 30 天被谁读过、影响过什么决策」答不出 → 删除候选。（= 原理一）
2. **有门禁的机制存活，纯自觉的衰减** ⚠️：维护动作必须在关键路径上（git hook / CI），否则制度再完美也形同虚设。**门禁 ≈ 让机器当那个消费者。**（= 原理二；旧版给它配季审是错的，见 §7）
3. **做了 ≠ 状态同步** ⚠️：「我翻了开关」被当成「全链路已切」。漂的总是「描述现实的状态」（有保质期），不是「结构」（昨天对今天还对）。判断「X 切了没」，永远列出 X 所有入口逐一验证，不能凭「我做过切换动作」。
4. **散落导致重复** ⚠️：真相散落 + 一个模糊数字被当成全量 → 靠记忆拼凑 → 拼错 → 重复劳动 → 不回写权威 → 散落依旧（恶性循环）。破法 = 动手前核实 + 单一权威。（= 原理三）
5. **不知道少了什么，比已知少了什么更危险** ⚠️：一个写死的过滤让大批数据长期没人发现（无 TODO、无触发条件、无自检）。「已知不全」总会上 backlog 补；「不知道不全」永远不补。**推论**：要定期覆盖度对账，不靠一次性导入。
6. **主操作方向决定结构** ⚠️：照搬一个 query-heavy 的知识库方法论，套到 ingest-heavy（对话沉淀）场景，会让每轮变成大动作、不可持续。**看框架名字照搬 vs 审主操作方向，是常见陷阱。**
7. **治理重认知不重形式** ⚠️：行数预算 / one-in-one-out 是为人脑记忆设计的；对 AI 重「认知是否清晰、机器是否真拦得住」，别为「行数零净增」删有用内容或耗一整轮凑数。（= 原理四；新架构据此砍掉了形式机制本身）
8. **知识由 AI 编译，而非人工整理** ⚠️：知识库 AI 在 ingest 时自主编译生长、面向 AI 消费（宁全勿简），人只输入 + 决策。最大的坑是把「精简到人脑可记」套到知识库（与 karpathy 原意相反）；另一个坑是讨论「沉淀机制」的会话本身零沉淀。
9. **声明式解析优于缓存状态** ⚠️：「有 X 用 X 否则用 Y」声明成 fallback 链 + 读时解析，别预存「赢家」再同步——写时缓存的隐藏成本是失效（数据变要重算、漏则 stale、重算有 bug）。读时解析数据变下次读自动反映、零同步。（软工对应「物化视图 vs 视图」）（= 原理三）

**反模式（Do NOT）**：

- ⚠️ **用「加治理」治「治理没被用上」**：每次发现规则没被遵守就再加一条规矩——下次又不读，越治越重。**缺的不是规矩，是「动手前真去核实」这个动作；动作，加文档改变不了。** 正解：改行动入口（必读文件的凌驾铁律）+ 机器拦截，不是加资料。
- ⚠️ **治理为治理生产证据**：体系长到几万行治理文本，精确度量机制存活率，却从没度量「为产品防止了几次事故」。信号出现 → 正确动作是**冻结治理面、只留门禁运转**，不是再加机制。（= §7 第二幕的病）
- ⚠️ **改枢纽文件不查引用方**：级联失效、引用方变孤儿。改前 grep 所有引用方。
- ⚠️ **回写端有、读端没接**：飞轮空转。回写后立即 grep 消费方是否真查表。
- ⚠️ **临时 flag 写代码无 TODO**：永远没人想起删（见定律 5）。
- ⚠️ **为凑数字调措辞**：言之有物的决策沦为装饰。
- ⚠️ **新建子目录溜进门禁盲区**：门禁覆盖面差一层，新结构没被守。门禁要覆盖到「功能分区」层级。

## 9. 最诚实的一条（边界，别忽悠自己）

- **治理无法 100% 消除认知错**——这是有智能体在环的系统的固有属性。**第一次踩新坑消不掉**（AI 凭已有上下文快速行动是固有倾向）。
- **能做到的是**：同一个坑踩第二次被机器拦 + 状态漂移机器报警 + 结构形式机器守住。能根治的是「反复掉同一个坑」，不是「免疫所有坑」。
- **最终兜底永远是人在环**——人的「当场质问」是唯一能抓机器测不了的语义漂移的门禁。目标是让人越来越少需要质问（坑被机器拦在前面）。
- **治理自身要防过度**——这是 §7 那个故事的全部教训：**越过收益递减点后，继续加治理就是最大的反模式。架构够了就停，转向用它、验证它**（跑 evals），而不是继续加它。会得这个病的，恰恰是那些「定律写得最清醒」的体系——因为它们最擅长造精巧的元层。

## 附：只能选 5 条带走

1. **把治理当成本，不当资产**——每条规则默认是负债，由真实失败挣来才留；架构够了就冻结治理面、转去用它验它。
2. **为消费者而生，无消费者即废**——没人因它改行为的标记必衰减成装饰；治理的消费者是产品，不是治理自己。
3. **腐烂默认，靠持续自动而非周期意志**——校验绑进每轮 hook / CI（机器当消费者），别靠「记得定期审查」（季审必死）；纠错改成事件驱动（lint 报错 / evals 答不出 / 用户打脸 / 撞见即改）。
4. **增量进唯一真相（SoT）+ 为 AI 写**——只记 delta、归档进生长的 SoT、一处权威不抄副本；入口文件求「最小高信号 token 集」，知识库才宁全勿简。
5. **用 AI 验 AI + 知道定律≠不犯定律**——拿无记忆 AI 跑 evals 当客观判据（口头保证不可信）；你完全可能一边写下定律、一边亲手违反它，所以要定期拿冷 AI 把自己重新量一遍。
