# 门禁判定面最小化:判「改什么」看目标,不做全文匹配

- **来源**:尾货处理中心(clearance-center)· 2026-07-19 · v3 protectedPaths 首次启用当场误伤实证(机器实证;负责人批准 promote)
- **场景**:v3 模板 pre-tool-use hook 的保护路径判定 = 把整个工具输入序列化后做子串匹配。项目把「仅负责人可改」的章程文件设为保护路径后,第一次写别的文档(内容含指向章程的链接)即被误拦;Bash 分支同理会拦纯读命令(cat/grep)甚至 commit message 里出现的文件名。
- **执行者做法**(模板):匹配面 = 整个输入,宁滥勿漏。
- **负责人要的做法**:判定面收敛到与风险语义对应的最小字段——Edit/Write 只比对目标路径字段;Bash 只拦写入形态(重定向/tee/sed -i/mv·cp 目标位),纯读放行;字符串防线拦不住的旁路(解释器写文件等)如实披露,由上层规则与人工责任兜底。
- **可泛化边界**:一切字符串防线类门禁。机制与项目无关:被保护标识符必然会被合法引用(文档链接、注释、commit message),匹配面越大误伤率越高;而**误伤率决定门禁存活率**——高误伤门禁很快被人加白名单、习惯性 `--no-verify` 或整个拆除,最终提供的保护归零。「宁可误拦」在门禁设计里是自败策略,不是保守策略。
- **载体**:v3.1.2 模板修复(pre-tool-use.mjs 判定面最小化 + 3 条误伤回归断言);个人知识库同名判例(promoted,证据链)。
