# 机器节拍与数字门槛不继承人类惯性

- **来源**:产品中心(母版)· 2026-07-17 · 同日两例同族,负责人两次当场纠正
- **场景**:①AI 为「判例提炼进 playbook」设了「≥2 个项目出现」的数字门槛;②AI 给纯机器执行的知识库语义 lint 排「每周一 09:00」节拍。
- **执行者做法**:沿用为人设计的形式(可数门槛、周报节拍),因为「看起来像制度」。
- **负责人要的做法**:①「一个项目提炼出来的东西就不好吗?你这个形式化是哪里来的?」——升级判据 = 因果机制是否普适,不数出现次数;②「有了 API 为什么不每天?非要问题越积越多?」——机器维护成本趋近零,日结不积压。
- **可泛化边界**:节拍与门槛跟着**消费者/执行者**走——执行者是机器/LLM,频率按成本与时效算、准入按机制分析;消费者是人,才按注意力预算设计(如「待拍板清单 ≤10 项」保留,因为读者真的是人)。周报、季审、行数预算、可数门槛这类形式脱离服务对象仍会自我复制,是形式主义高发机制。等第二个项目再掉一次坑才收编,违背 playbook 存在的目的。
- **载体**:已下沉全局会话协议第 7 条(负责人本机,每会话注入)+ 个人知识库同名判例(证据链,负责人本机)。
