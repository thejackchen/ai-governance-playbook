# 多人协作仓开工先 fetch 对账,防双线分叉

- **来源**:产品中心(母版)· 2026-07-17 · 全项目审计实证后负责人批准预案收敛
- **场景**:仓库有第二位协作者(同事维护商城 Catalog V2)。本地线连续 10 天只在本地 commit(治理 v2 + 三个决策),从未 fetch/push;同事在 origin 上开发并于 07-15 生产部署了整套新 API。两线互不知情。
- **执行者做法**:每轮会话直接在本地 main 干活并宣布「已完成」,从不与远端对账;治理心跳(GitHub cron)因 workflow 文件只在本地而从未真正跳过,且无人察觉。
- **负责人要的做法**:审计暴露后批准「先出合并预案 → merge 保双线历史 → 全套验证绿 → push」收敛;防再发要求机器化。
- **可泛化边界**:①多人(或多线程)协作仓,**每次开工先 `git fetch` + ahead/behind 对账**,可下沉 SessionStart hook 自动回显;②「已完成」的声明以**推送到共享分支**为准,本地孤岛上的完成不算完成;③依赖默认分支的机器载体(GitHub scheduled workflow、CI)在文件推上去之前等于不存在——「本地跑通」≠「远端会跳」,装完必须远端实证一次;④长期分叉的合并优先 merge 保历史,勿 rebase 改写。
- **载体**:待下沉——SessionStart hook fetch 对账(产品中心 B1 批次);心跳对账包加「距上次心跳天数」自检。
