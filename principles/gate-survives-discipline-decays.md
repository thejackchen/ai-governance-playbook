# 有门禁存活·纯自觉衰减（Gate Survives, Discipline Decays）

> 绑了机器门禁（hook / CI / cron）的机制会存活；纯靠自觉执行的机制会衰减。

## 为什么成立（机理）

执行者（尤其是 AI，但人也一样）在压力 / 遗忘 / 上下文切换下，会自然跳过「没有强制点」的步骤。一条规则若只写在文档里、靠「记得做」，它的执行率随时间单调下降——不是因为谁懒，而是因为没有任何力量在「不做」时产生后果。反之，挂在**关键路径**上的门禁（提交时跑的 hook、push 时跑的 CI、定时触发的 cron），把「执行」变成「绕不过去的副作用」：不满足就过不了。门禁的本质，是**让机器替你当那个永不疲倦、永不遗忘的消费者**（见 `signal-needs-consumer.md`）。

## 踩坑实例（⚠️）

⚠️ 同一个项目里做过一组对照：凡是绑定了机器门禁的机制（如「行数预算」靠 lint 在提交时检查），全部存活、持续生效；凡是纯靠 AI 自觉执行的维护仪式（如「定期自我审查某类资产是否该毕业 / 归档」），全部衰减——其中一个仪式断档三周无人执行，且没有任何信号提醒它断了。

**why**：自觉仪式没有「不做就报错」的机制，断了也悄无声息。而且 AI 会信誓旦旦地说「我跑了检查，结果 OK」——但其实根本没跑（LLM 输出不可信）。所以关键检查停在「自觉」级，等于没有。

**推论**：维护仪式必须落在关键路径上（git hook / CI / cron）。制度设计得再完美，没有执行点，就形同虚设。

## 可迁移适用场景

- 任何「我们约定以后都要……」类的团队 / AI 规范：没有 CI 检查的约定，半年后基本失效。
- 代码规范、提交信息格式、文档同步、依赖更新——靠 review 时「记得提醒」远不如 lint / hook 强制。
- 定期任务（轮换密钥、清理过期数据、跑覆盖度对账）：写进文档不如挂 cron。
- 任何「AI 自我汇报已完成」的环节：必须有独立的机器验证，不能信它的自述。

## 怎么机器化

**这条本身就是关于机器化的元定律**，可执行性极强：

- **机制出生登记制**：新增任何强制规则 / 仪式前，先登记它的门禁是谁——lint 编号 / pre-commit hook / CI / cron / （或显式标注「纯自觉，接受衰减」留案底）。登记不出执行点的强制规则，当场拒收。
- **强制等级分层**（决定一条规则多可靠）：
  - L1 自觉：AI 自己记得做 —— ⚠️ 不可靠，会衰减，需定期清理（无执行证据就降级 / 删）。
  - L2 pre-commit hook：本地拦，可绕过（风险自负）。
  - L3 CI：push 时强制，不可绕。
  - L4 人审：拍板类，靠人在环。
- **元门禁（守护守护者）**：让「存在未登记的强制规则 / 检查」本身成为机器可检的违规——lint 验证所有检查、所有政策段落都在注册表有账。
- 关键检查必须落到 L2 / L3（机器强制），绝不停在 L1。

> 人在环（L4）是唯一能抓机器测不了的语义漂移的兜底门禁，但目标是让坑尽量被前面的机器门禁拦住，人越来越少需要当场质问。
