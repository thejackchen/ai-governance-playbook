# 主操作方向决定结构（Operation Shape Determines Structure）

> 选架构 / 方法论，要看你的**主操作方向**（读多还是写多），而不是照搬一个名字相同、方向相反的成熟框架。

## 为什么成立（机理）

同一套数据结构（一个索引、一份日志、一张表），在「读多写少」和「写多读少」两种负载下，扮演的角色和承受的代价**完全不同**。一个为「偶尔写入、频繁查询」优化的设计（写时多花点功夫建好索引，让后续无数次查询都快），套到「频繁写入、偶尔查询」的场景上，会让**每一次写入都变成一个大动作**——结果是不可持续：每一轮都要付出沉重的整理代价，很快就维护不动。框架的「名字」（知识库、日志、索引…）会诱导人照搬它的全套做法，但决定做法是否合适的，是你的**主操作方向**，不是名字。

## 踩坑实例（⚠️）

⚠️ 照搬了一套面向「查询为主」场景（如学术论文库：写入一次、被检索无数次，所以写入时舍得花大力气建全套交叉引用）的知识库方法论，套用到一个「**写入为主**」的场景（如持续的对话沉淀：每轮都在写，查询相对少）。结果每一轮沉淀都被要求做成大动作（一次更新十几个文件、维护全套交叉引用），很快变得不可持续。

**why**：两个场景的主操作方向相反。论文库优化「写一次、读千次」，所以写时重投入是划算的；对话沉淀是「写千次、读偶尔」，把写做成重动作就是灾难。**看框架名字照搬 vs 审主操作方向，是常见陷阱**。同一个 index / log，在两个方向下角色完全不同。

## 可迁移适用场景

- 选数据库 / 存储引擎：读密集（适合重索引、物化视图）vs 写密集（适合 append-only、轻索引）。
- 缓存策略：读多写少才值得激进缓存；写多的话缓存失效成本可能反超收益。
- 文档 / 知识管理流程：高频产出场景，别用低频精修场景的「每次都全量整理」流程。
- 借鉴任何「最佳实践 / 成熟框架」时：先问它原本是为什么操作方向设计的，和我的一致吗？
- API 设计：读多的接口和写多的接口，分页、批量、一致性的取舍完全不同。

## 怎么机器化

**这条主要靠设计判断，机器难以直接拦**——它是「选型决策」层面的认知，不是可被 lint 检出的代码模式。

可落地的是把它变成**决策清单 / 设计评审的一道必答题**：

- 在引入任何外部方法论 / 框架 / 架构模式前，强制回答：「我的主操作方向是读多还是写多？这个框架原本为哪个方向设计？两者一致吗？不一致的话，我要改哪些做法？」答不上来不准照搬。
- 配套一个**分级策略**抵御「写动作过重」：把高频写场景的沉淀做成轻量动作（每次只碰 1–3 个必要文件），只在低频的里程碑时刻才做重整理（全量交叉引用）。别把高频场景按低频框架的「每次大动作」来做。
