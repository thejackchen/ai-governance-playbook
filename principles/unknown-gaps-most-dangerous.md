# 不知道少了什么，比已知少了什么更危险（Unknown Gaps Are the Most Dangerous）

> 「已知不全」总会上待办、迟早补上；「不知道不全」永远不会被补——因为没人知道要补。

## 为什么成立（机理）

补一个缺口的前提是「知道它存在」。已知的缺失（你清楚少了某块）会自然进入待办清单，被某次迭代消化掉——它有可见性，就有被解决的路径。而**未知的缺失**——一个静默生效、无人察觉的限制 / 过滤 / 默认值——没有任何东西指向它：没有 TODO 标记它、没有触发条件让它冒头、没有自检周期会碰到它。它就这样无限期地、安静地扭曲着系统的行为，而所有人都以为一切正常。危险不在于「缺」，而在于「缺得没有信号」。

## 踩坑实例（⚠️）

⚠️ 一个早期为了快速上线（MVP）而写死的过滤条件，悄悄把一大批数据（量级达十几万条）挡在了系统之外，**半年无人察觉**。它没有 TODO 注释、没有任何触发条件会让它暴露、也没有任何周期性自检会撞上它——所以它就那样静默生效，所有人都以为数据是全的。

**why**：这个过滤是「未知缺口」的标本——它从存在的第一天起就没有任何指向它的信号。临时 flag 写进代码却不留 TODO，是制造未知缺口的经典方式：永远没人想起删它。「已知不全」会上 backlog 总会补；「不知道不全」永远不补。

**推论**：对抗未知缺口，靠的不是一次性的「导入 / 初始化做对」，而是**定期的覆盖度对账机制**——主动地、周期性地拿真相去核对系统，让静默的缺口现形。

## 可迁移适用场景

- 任何「临时 / MVP 阶段写死」的限制、过滤、上限、默认值——上线后没人记得它还在。
- 分页 / 查询默认 limit 静默截断结果，调用方以为拿到了全集。
- 权限 / 可见性规则默默过滤掉一类数据 / 用户，无人发现某类对象「从来没出现过」。
- 同步 / 导入任务静默跳过一部分记录（异常被吞、条件不匹配），总量看着正常。
- 监控只覆盖「已知会出问题」的指标，对「从没想到要监控」的维度完全失明。

## 怎么机器化

**核心机制是「主动对账」，可周期化**：

- 建**覆盖度对账机制**并挂 cron / 季审：周期性地拿一个独立的真相来源去抽样核对系统的全集（如「实物抽样 vs 数据库」「上游总量 vs 下游入库量」「应有集合 vs 实际呈现集合」），差异超阈值告警。这是把「未知」逼成「已知」的唯一可靠手段。
- **禁止无 TODO 的临时 flag**：可用 lint 检查代码里「写死的过滤 / 上限 / 默认 limit」是否带显式标记（TODO + 触发条件 + 复查日期），无标记的硬编码限制报警。
- 对所有「截断性」操作（分页、limit、过滤）要求显式声明全集大小，便于调用方校验是否被静默截断。

**只能靠人 / 设计意识的部分**：要监控「自己根本没想到的维度」是逻辑上不可能由现有清单覆盖的——这正是为什么需要**人在环 + 冷上下文季审**（由无本会话记忆的人 / agent 重新审视「我们是不是漏看了某整类东西」），它能抓到机器和当班者都视而不见的盲区。

> 一次性「做对」治不了这条，因为静默缺口会随系统演化不断新生。必须把对账变成**周期性门禁**——呼应 `gate-survives-discipline-decays.md`：纯靠「记得检查」必衰减。
