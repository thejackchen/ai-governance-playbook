# 散落导致重复（Scattered Truth Breeds Repetition）

> 真相散落多处 + 一个被误解的模糊数字 → 动手前靠记忆拼判断 → 拼错 → 重复劳动 → 结论又不沉淀回权威处 → 散落依旧。这是个恶性循环。

## 为什么成立（机理）

当同一件事实的「权威答案」不在一个确定的地方，而是散落在多个文件 / 聊天记录 / 人的脑子里时，任何要用到它的人 / AI 都被迫**在动手前临时拼凑**一个判断。拼凑就会出错——尤其当中混入一个**语义被误解的数字**（典型：一个「子集覆盖率」被当成「全量覆盖率」）。错误判断导致做了不该做的事或重做已做过的事（重复劳动）。而最致命的是闭环的最后一环：拼出来的结论 / 踩到的坑，**没有被沉淀回一个权威位置**——于是下一个人 / 下一轮 AI 面对的依然是散落的真相，循环重启。每转一圈，散落不减反增。

## 踩坑实例（⚠️）

⚠️ 一个关键事实散落在多处，且伴随一个模糊数字——比如「覆盖率 96%」，但这个 96% 其实只是某个**子集**的覆盖率，被普遍当成了**全量**覆盖率。结果有人据此判断「缺口不大，不用再补」或反过来「还差得远，再跑一轮采集」，而真相要靠把多处信息拼起来才看得清。拼错了，就白做一轮。

**why**：① 真相没有单一权威落点，逼着每个人靠记忆 + 局部信息拼。② 数字的语义（子集 vs 全量）没有和数字本身绑定记录，极易误读。③ 拼出的结论又没回写到权威处，所以同样的误判会反复发生。

## 可迁移适用场景

- 任何「这个指标到底什么口径」反复被问、反复有人答错的场景（DAU 含不含内部账号？转化率分母是谁？）。
- 同一份配置 / 参数在多个文件里各写一份，改了一处没改其余。
- 「上次那个结论是什么来着」要翻聊天记录 / 多个文档才能拼出来。
- 跨团队协作里，「真相之源」没指定，每个团队维护自己的一份，逐渐分叉。
- 数字脱离其口径定义单独流传（截图里一个百分比，没人记得它怎么算的）。

## 怎么机器化

**部分可机械化**：

- 为每类关键事实**指定唯一权威源（SoT）**，其余位置标注「派生自 X，勿直接改」；可用 lint 检查是否有「应派生却被就地硬写」的副本。
- 数字 / 指标**永远和口径定义绑定存储**（同一处写明分子分母 / 子集还是全量），禁止裸数字流传。
- 断链检查、索引完整性检查：保证「权威源」可被找到、不成孤儿。

**核心其实是流程门禁 + 文化，机器只能辅助**：

- **动手前铁律**（对本定律的直接破法）：任何有成本的操作前，强制三问——① 这事 / 这判断之前做过没？（搜历史，别重做）② 当前真实状态？（查数据 / 代码实证，不凭「我以为」，也别拿残缺 grep / 截断输出当全貌）③ 权威记录在哪？查不到就**先建 / 补**，别让结论只活在聊天里。
- 关键是闭环最后一环：**每次拼出新结论 / 踩到新坑，必须回写权威处**——否则循环不断。

> 这条与 `done-is-not-synced.md` 共享同一个解药（动手前核实真相、非记忆）。「不知道少了什么」的隐性缺口（见 `unknown-gaps-most-dangerous.md`）往往就藏在散落的真相里。防散落的根本，是把结论**编译进单一权威**而非散落各处——见 `knowledge-compiled-by-ai.md`。
