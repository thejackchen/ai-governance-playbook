# 声明式解析优于缓存状态（Declarative Resolve over Cached State）

> 「显示什么 / 用哪个值」这类决策，声明成**规则（fallback 链 / 优先级）+ 读时解析**，而不是预先算好「赢家」存起来再想办法同步。**读时解析让「状态同步」这个负担直接消失。**

## 为什么成立（机理）

任何「有 X 用 X，否则用 Y」的决策有两种实现：

- **写时缓存**：预先按规则算出赢家、写进字段（物化），数据一变就要触发重算 / 失效。
- **读时解析**：不存赢家，读取时按规则现算（`COALESCE` / fallback）。

写时缓存的隐藏成本不是「算」，是 **失效（invalidation）**——数据变了要记得重算，漏一处就 stale；重算逻辑本身会有 bug。读时解析没有「赢家状态」这个东西，**自然没有要同步的东西**：数据变了，下次读自动反映。

## 踩坑实例（⚠️）

⚠️ 把「赢家」写时缓存（按来源优先级算出主值存字段），后来全量重算时**把高优先级的值翻回了低优先级**，导致已升级的优质数据批量退化——失效 / 重算逻辑的典型 bug。读时解析（`COALESCE` 优先级链）不会有这类「重算翻车」。

⚠️ 数据量大时，写时缓存让人焦虑「这么多条怎么保证都同步」——**这个焦虑本身是信号：你在维护一个本不该存在的状态**。

## 软件工程里的同款

- **物化视图（要 `REFRESH`）vs 普通视图 / 计算列（读时算、自动最新）**——最直接对应。
- **schema-on-read vs schema-on-write**（数据湖）。
- **fallback 链 / `COALESCE`**（声明式优先级）。
- **CQRS 物化读模型**——写时缓存的赢家就是一种 read model，要靠事件同步。

## 电商里的同款

- 商品**图片** = SKU 图 ?? SPU（系列）图 ?? 默认图——「单品图 ?? 系列图」就是它，电商标准做法。
- 商品**最终价** = 促销价 ?? 会员价 ?? 原价（促销常变，多用读时解析）。
- 库存 / 可售状态同理。

## 更优方案谱系（按规模选，别一步到位）

| 方案 | 自动最新 | 同步负担 | 适用 |
|---|---|---|---|
| **纯读时解析** | ✅ | 无 | 默认起点；读不是极高 QPS |
| 读时为主 + 热点写缓存 + **短 TTL** | 近似 | 低（TTL 到期自动刷，不用精确失效） | QPS 高、热点集中 |
| 事件驱动失效 | ✅ | 中（要事件基础设施） | 强一致 + 高 QPS |
| 增量物化视图 / CDC | ✅ | 高 | 超大规模、专门团队 |

**判据**：先用纯读时解析（零同步负担）；性能真不够，优先「轻缓存 + 短 TTL」而非「精确失效」（TTL 比追踪每个数据变化简单一个量级）；别一上来就 CDC（过度）。

## 怎么落地

- 决策规则**声明在一处**（fallback 链 / 优先级配置），不在各消费端硬编码。
- 解析放查询层（API / view / `COALESCE`），各端不重复实现。
- 缓存**只在性能真不够时加**，且加缓存 = 显式接受失效负担（默认不加）。

> 关联：写时缓存的「赢家状态」要同步，正是 [`done-is-not-synced.md`](done-is-not-synced.md)（状态会漂、要同步）的负担来源；规则声明在一处、读时算，呼应 [`cognition-over-form.md`](cognition-over-form.md)（认知清晰 > 形式完备）。
