# 做了≠状态同步（Done Is Not Synced）

> 「我执行了某个动作」不等于「系统状态已全链路改变」。判断状态，永远不能凭「我做过那个动作」的记忆。

## 为什么成立（机理）

一个动作往往只作用于系统的**一个入口 / 一个副本**，而真实的「状态」分布在多个入口、多个消费端。把「我翻了开关」当成「全链路已切换」，是把**局部动作**误当成**全局事实**。更深一层：会漂移的永远是「**描述现实的状态**」——它的正确性有保质期，现实往前走，它就自动过期；而「**结构**」（昨天对今天还对的真理）不会这样漂。人 / AI 的记忆记的是「我做了 X 这个动作」，但要回答的问题是「X 这个状态现在覆盖了多少」——这两者之间隔着「所有入口是否都被覆盖」的客观事实，记忆填不了这个缝。

## 踩坑实例（⚠️）

⚠️ 「翻了一个开关 / 改了一处配置」被当成「整条链路都切过去了」。但那个开关只对接了链路的**一个入口**，其余入口仍走旧路径。事后凭记忆断言「已经切了」，与现实不符——真相是只切了一部分。

**why**：动作的作用域 ≠ 状态的作用域。一个动作天然只覆盖它直接对接的入口；而「切了没」问的是所有入口的覆盖面。凭「我做过切换动作」回答覆盖面问题，必然高估。

**推论**：判断「X 切了没 / 改全了没 / 上线了没」，永远不能凭「我执行过那个动作」，必须**列出 X 的所有入口 / 所有消费端，逐一验证覆盖**。

## 可迁移适用场景

- 配置 / 特性开关：改了一处，但代码里有多处读取点（多个服务、多个环境、缓存层）。
- 数据迁移 / 改名：改了主表，但派生表、缓存、索引、下游导出未同步。
- 「部署完成」：发了一个服务，但多副本 / 多区域 / CDN 缓存未全覆盖。
- 权限 / 密钥轮换：在一侧生成了新凭据，但另一侧未激活，或旧凭据未吊销。
- 文档 / 状态记录：改了代码，但描述现状的文档没跟着改（状态线漂移）。

## 怎么机器化

**部分可机械化**：

- 对「描述现实的状态」类记录设**新鲜度门禁**：若状态文件落后于实际变更流水超过 N 天，lint / CI 报警（状态线漂移检测）。
- 对有限且可枚举的入口（如所有读取某配置的代码点），用 grep / 静态分析列出全集，做成「覆盖面清单」，切换时逐项核对、机器校验是否全绿。
- **回写端有、读端没接**的飞轮空转，可机器检：回写后立即 grep 消费方是否真的查了那张表 / 那个字段。

**只能靠协议 + 人在环的部分**：「列出所有入口」这步本身需要认知，机器列不全。所以配一条**动手前铁律**作为流程门禁——执行任何有成本的切换前，强制核实：① 这事做过没？② 当前真实状态是什么（看数据 / 看代码实证，不是「我以为」）？③ 有无单一权威记录？这是对本定律的直接防御。
