# 信号要有消费者（Signal Needs a Consumer）

> 一个有维护成本的标记 / 指标，若没有任何决策或行为真的读它、因它改变行为，它**必然衰减成装饰**。

## 为什么成立（机理）

任何标记、评级、状态字段、指标都有持续的维护成本（要打、要更新、要保持准确）。维护它的动力，只能来自「它被消费」——有人 / 有机器读它，并据此**改变行为**。一旦「写」与「行为改变」之间的链路断了，维护就变成纯负担，理性的执行者（人或 AI）会自然地少写、漏写、敷衍写。它不会立刻消失，而是缓慢腐烂成一个**看起来还在、其实没人当真**的装饰品——这种「半死不活」状态比直接删掉更危险，因为它仍在消耗注意力、还可能被误当成真实信号。

## 踩坑实例（⚠️）

⚠️ 给「沉淀的知识条目」设计了一套成熟度评级（如三级制：草稿 / 验证中 / 已固化）。结果几十条条目里只有极少数真的有验证记录——因为 AI 在引用这些知识时，**从不根据成熟度等级改变自己的行为**（不会因为「这条只是草稿」就更谨慎地用它）。评级字段于是全面荒废。

**why**：评级的「消费者」从未真正存在。设计时假设「标了等级，引用方就会区别对待」，但没有任何机制（人或机器）强制引用方读这个等级、因它改判。没有消费者的信号，打得再勤也是自欺。

**可机械检测的装饰化判据**：对任一信号问一句——「最近 N 天它被谁读过、影响过哪个决策？」答不出来，就是删除候选。

## 可迁移适用场景

- 任何「优先级 / 严重度 / 置信度 / 成熟度」标签：若下游流程不因标签分流，标签就是装饰。
- 仪表盘上「没人看、看了也不据此行动」的指标。
- 代码里的 `@deprecated` / TODO 注释、文档里的「状态：草稿 / 已废弃」——若无机制据此触发清理或区别对待。
- 流程中要求填写、但后续从不被任何审批 / 路由读取的字段（工单分类、风险等级…）。
- 任何「为了好看 / 为了显得严谨」而加、但没有消费端的元数据。

## 怎么机器化

**可机械化（人在环辅助）**：

- 给每个受治理的信号登记一个「消费者」字段：谁 / 哪条规则 / 哪个流程读它。**登记不出消费者的信号，不准出生**。
- 定期（如季审）自动扫描：列出「过去 N 天无任何读取 / 无任何决策引用」的信号 → 进删除候选清单。
- 若信号有访问日志 / 引用可追踪（被哪个文件、哪次决策引用），可写脚本统计引用计数，零引用自动告警。

**只能靠人 / 季审的部分**：「读了之后是否真的因它改了行为」这种语义判断，机器测不准，需在季审时人工抽查或用行为评测（让 AI 在有 / 无该信号下做同一决策，看输出是否不同）。

> 让机器当那个消费者，是把「自觉维护」变成「不维护就被拦」的最稳办法——见 `gate-survives-discipline-decays.md`。
