# AI 治理方法论库 · AI Governance Playbook

> 一套**可迁移的 AI 项目治理方法论**——从真实项目一线实战萃取的定律、模式、反模式、踩坑教训。
> 给「用 AI 维护的项目」借鉴。**核心原则：借思想，别抄实现。**

## 这是什么 / 不是什么

| 是 | 不是 |
|---|---|
| 可迁移的治理**思想**（定律 / 模式 / 反模式 / 架构框架） | ❌ 一套强制套用的治理实现 |
| 可 fork 的起步**脚手架模板** | ❌ 现成的 lint 代码 / 文件直接复制 |
| 多项目**共建**、越积越厚的踩坑教训 | ❌ 凭空的理论（每条定律都是真踩出来的） |

为什么不抄实现：每个项目的使命、技术栈、团队、阶段都不同，治理**实现**必须按自己的来长。可迁移的只有**思想**。

## 两层心智模型（先建立这个）

治理 = **两层叠加，缺一不可**：

| 层 | 管什么 | 在本库哪 |
|---|---|---|
| **① 行为层**（micro · 每个任务怎么做） | AI 单次工作的行为铁律：动手前核实 / 简洁防过度 / 精准修改 / 目标驱动 / 验证后宣称 | [`starter/behavior-baseline/`](starter/behavior-baseline/)（`CLAUDE.md` 给 Claude Code、`AGENTS.md` 给 Codex） |
| **② 系统层**（macro · 项目怎么治理） | 项目级治理体系：三线一底座 / 定律 / 机制 / 反模式 | [`governance-architecture-essence.md`](governance-architecture-essence.md) + [`starter/`](starter/) 治理脚手架 |

> **行为层是底座**（任何项目、任何任务都该有，跨工具）；**系统层是上层建筑**（项目级治理，规模大了才需要）。两层正交、叠加、不冲突。新项目可只从行为层起步，长大了再上系统层。读本库**两层都该看**。

## 为什么存在：跨项目的「踩第二次就拦」

单个项目内，治理让同一个坑踩第二次被拦。这个库把它升一维——**一个项目踩的坑、得的定律，汇到这里，你所有项目的 AI 启动就知道、不用再踩。** 单项目防重复 → 跨项目防重复。这是复利。

## 结构

```
├── governance-architecture-essence.md   ★ 核心精华：一次领悟整套架构（三线一底座+定律+机制+反模式）
├── principles/                          可迁移定律，一条一文件（便于多项目共建追加、单条引用）
├── starter/                             可 fork 的脚手架模板（北极星 / 驱动引擎空壳 + 填写指引）
└── HOW-TO-USE-FOR-AI.md                 ★ 别的 AI 怎么用（提示词模板）
```

**先读哪个**：想快速领悟整套 → `governance-architecture-essence.md`；想让你的 AI 用起来 → `HOW-TO-USE-FOR-AI.md`；查单条定律 → `principles/`；起新项目 → `starter/`。

## 怎么共建（回流）

你的某个项目踩出一条**可迁移**的新定律 / 反模式 → 让该项目的 AI 在 `principles/` 加一条（一个文件，格式参照已有的）。

判断标准：**可迁移的才回流**（换个项目也成立的定律/模式）；项目特定的（具体规则/数值/业务）留在本项目，不要污染这个库。

## 一条这库自己也守的铁律

**治理重认知不重形式 + 防过度。** 这是个**轻知识库**（markdown 思想为主），不是一个要复杂 CI / 同步机制去重治理的工程。⚠️ 别让它自己变成负担——那就违背了它自己写的反模式「治理为治理生产证据」。
