# 如何让你的 AI 使用这套治理方法论

> 给「想用 AI（Claude 等）帮项目做治理」的人。把下面对应场景的提示词发给你的 AI，它就能借鉴这套方法论。

## 先给你的 AI 立一条规矩

**借思想，别抄实现。** 这套方法论的价值是可迁移的**定律 / 模式 / 反模式**，不是它的具体规则、文件结构、lint 代码。你的项目治理实现，必须按自己的使命、技术栈、团队、阶段来长。

---

## 提示词模板（按场景选一个）

### 场景 1 · 给新项目冷启动治理

```
读这个治理方法论库的 governance-architecture-essence.md 和 principles/。
基于它为我的项目设计一套【轻量】治理架构：
- 用「三线一底座 + 横切关注点」框架（治理线/状态线/沉淀线 + 执行底座 + 横切）
- 起一个北极星：fork starter/CONSTITUTION.template.md，填我项目的使命和不变真理
- 关键规则必须带执行点（lint / git hook / CI），不准建纯自觉的（纯自觉必衰减）
- 【不要照抄它的实现】，按我项目的 [技术栈 / 团队规模 / 当前阶段] 落地
先给我方案，我拍板再动手。
```

### 场景 2 · 审视现有项目的治理

```
读这个治理方法论库的 governance-architecture-essence.md（重点 §5 定律 + §7 反模式）。
用这些定律审视我项目现有的治理，找出：
- 哪些规则是纯自觉（会衰减）、该机器化
- 哪些「信号没有消费者」（没人因它改行为 = 装饰，可删）
- 有没有踩反模式：治理为治理 / 散落导致重复 / 回写端有读端没接 / 用加治理治"治理没被用上"
给我一份对照清单 + 优先级，先别改。
```

### 场景 3 · 回流共建（你的项目踩出了新教训）

```
我的项目踩出一条治理教训：[在这里描述发生了什么、踩了什么坑、得到什么结论]。
先判断：它是「可迁移的定律/反模式」还是「项目特定的」？
- 如果可迁移：在这个方法论库的 principles/ 加一条（一个文件，英文 kebab 命名，
  含「一句核心 + 为什么 + 踩坑实例 + 可迁移适用场景」，格式参照已有文件）。
- 如果项目特定（具体规则/数值/业务）：留在我本项目，不要回流污染这个库。
```

---

## 务必传达给 AI 的 4 条关键提醒

1. **借思想别抄实现** —— 换个项目，实现必须重长，思想才可迁移。
2. **治理要带执行点** —— 纯自觉的规则必衰减；让机器（lint/hook/CI）当那个"消费者"。
3. **治理防过度** —— 架构够了就停。继续加治理、让治理为自己生产证据，是最大的反模式。
4. **定律的价值在「踩坑 why」** —— 这套不是理论，是真踩出来的。每条定律带的 ⚠️ 教训才是精华，别只记结论、忽略它为什么成立。

---

## 一句话总结

把这个库当成一个**有经验的治理顾问的笔记**：它告诉你「什么会衰减、什么坑会反复踩、为什么」，但你的项目怎么治理，要你和你的 AI 自己决定。
