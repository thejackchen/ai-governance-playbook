# 治理自纠：事件 → 动作（取代季审）

> 纠错不靠「定期审查」（季审从没跑过 = 死触发器），而是绑到本来就会**自动 / 高频发生**的事件。事件一来，照表纠。这是 fitness functions 的精神：纠错嵌进工作流，不另设仪式。
>
> 这落地不变原理之一：**腐烂是默认，靠持续自动纠错托住，不靠周期意志**。

| 事件（自动 / 高频发生） | 纠错动作 |
|---|---|
| **lint 报 error / warn**（每次结束前、commit 时都跑） | 当场修到 0 error，别留到以后 |
| **[evals](governance-evals.md) 冷启动题某题答不出**（大改 / 换模型时，派无记忆 AI 跑一遍） | 改那道题对应的**承载文件**（修文件不修答案）——答不出 = 那块治理失效了 |
| **用户质问 / 打脸**（"这不对"、"你又附和了"） | 当场改 + 沉淀（踩坑进 `{踩坑沉淀目录:如 KNOWLEDGE_BASE/insights/}`、协作偏好进上手包） |
| **干活时撞见规则过时 / 自相矛盾** | 当场改（SoT 一处权威、只改一处）+ 记踩坑。**过时规则 = 有害，删不留**（错的指令比没指令更糟） |

每条都写清「谁触发、做什么」，且触发器本来就在工作流里跑——所以**不需要单独的纠错仪式或季审**。这就是「持续自动 > 周期意志」落到地上。

## 加规则 / 砍规则的判定

**加新规则 / 检查前，过三问**（四原理落地，过不了不加）：① 有具体消费者吗？（没消费者的规则别加——为消费者而生）② 是真实发生过的失败挣来的，还是预防想象出来的？（复杂度要挣来，想象的失败不挣）③ 机器拦得住吗——拦得住进 lint / hook；拦不住的别塞进 lint，靠 evals + 用户在环。

**砍规则的判定**：「没人用」= 没有任何文件链它（孤儿，grep 反链可判）+ 连续 ≥`{commit 阈值:如 50}` commits 无触碰、或干活时撞见即判 → 标候选、确认后砍。**用 commit 计数 / 事件触发，不挂日历周期**（日历周期就是被砍掉的季审）。过时 / 没人用的规则有害（错的指令比没指令更糟），删不留。

> 治理大改（动宪法 / `{接手手册:如 AGENTS.md}` / 砍加规则）后，跑一遍 [治理验收套件](governance-acceptance-test.md) 确认没把哪个环节弄坏。
