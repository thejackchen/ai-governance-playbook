# 执行铁律 — 动手前 / 动手时 / 宣称前

> [AGENTS 干活铁律段](../../AGENTS.md) 的完整正文。凌驾一切的工作纪律，根治「反复掉同一个坑」。
> 落地宪法（求真是命根 / 合伙人不附和 / 能简单绝不复杂 / 做完拿证据）。灵感来源：karpathy 编码准则（正文已自足，不依赖外链）。

补「启动 → 对话 → 结束」三协议缺的**执行**环节：协议管「读什么 / 响应什么 / 沉淀什么」，铁律管「动手的每一步怎么不犯低级错」。

## 动手前 — 三问，不凭记忆

执行任何**有成本的操作**（{列出你项目里高成本动作:如 抓取 / 切流量 / 批量改 / 发版 / 跑长任务 / 发凭据 / 删文件}）前，先核实：

1. **做过没？** 这事 / 这判断之前是否已做或已有结论 —— 搜 CHANGELOG / memory / 会话，别重做已做过的（{填一个本项目真实踩过的「重做已有工作」教训:如 重做了某个已存在的模块}）。
2. **真实状态？** 查数据 / 代码**实证**当前真相，不是「我以为」（切没切、覆盖多少、文件在不在、grep 截没截断）。
3. **权威记录在哪？** 查不到就先建 / 补，别让结论只活在聊天里。

**三问过不了，不动手。**

## 动手时

① **精准修改**：只动任务相关的代码 / 文件，不顺手碰无关的（改动范围 = 任务范围）。
② **目标驱动**：先定清「做完 = 什么样、怎么验证」，没有成功标准不开工。

## 宣称完成前 — 实证

用**实证**验证（跑测试 / 查数据 / 端到端），别拿残缺 grep、截断输出、旧缓存当全貌下结论，**尤其与刚做过的事矛盾时**。

## 衰减风险与兜底

铁律绝大多数是**语义自觉**，机器实时拦不住。按元定律「有门禁的存活、纯自觉的衰减」，它自身是高衰减候选 —— 兜底见 [治理自纠](governance-self-correction.md)：lint / evals / 用户打脸 / 撞见即改，事件驱动、非季审。
