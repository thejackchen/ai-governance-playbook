# 治理核心模型

> 本文件是方法论唯一权威。适配器只回答“在某个运行时怎么承载”，不得复制或发明第二套治理原则。

## 1. 治理是一条控制回路

一条规则只有走完以下链路才算进入治理：

```text
意图/风险 -> 可观察事件 -> 可判定条件 -> 载体动作 -> 证据 -> 事故反馈
```

- 文件保存意图和解释，不天然产生执行力。
- Hook 提供明确触发，但不一定具备可靠判定或阻断能力。
- CI 只有配合 required check / branch protection 才是真正闸门。
- IAM、只读凭据、数据库约束和 API 白名单通常比 CI 更靠近风险，也更强。
- AI 适合补充语义判断，不应成为确定性红线的唯一裁判。

## 2. 五个结构问题

治理只处理以下结构问题，或明确标为临时能力补丁：

| 编号 | 问题 | 典型载体 |
|---|---|---|
| S1 | 执行者会失忆 | 自动指令、状态投影、SessionStart |
| S2 | 人的意图不能自动传递 | 宪法、ADR、问题队列、人工批准 |
| S3 | 完成宣称不能自证 | 测试、证据、CI、状态探针 |
| S4 | 有些动作不可逆 | 权限、审批、PreToolUse、备份 |
| S5 | 注意力和维护预算有限 | Profile、规则预算、死亡条件 |

说不清解决哪个问题的规则，默认不进入核心；若只是当前模型或工具的缺陷，标记为 `[cap]` 并设置清理条件。

## 3. 载体必须同时描述两件事

### 触发时点

- `runtime`：运行时每次访问都生效，例如 IAM、schema、API 投影。
- `action`：工具调用或文件修改前后，例如 Hooks。
- `commit`：本地提交前，例如 pre-commit。
- `merge`：PR 或合并时，例如 CI required checks。
- `session`：会话开始、压缩、结束时。
- `schedule`：每日、每周或按周期自动触发。
- `manual`：只有人或 AI 主动读取时发生。

### 执行效果

- `impossible`：结构上无法违反。
- `block`：判定失败即阻止动作或合并。
- `approval`：必须由明确责任人批准。
- `warn`：不中断，但产生可见警告。
- `record`：只留证据，供追溯或审计。
- `document`：只提供解释，没有自动触发。

“L1 Hook”不足以说明执行力；必须写清它在什么事件触发、用什么条件判断、最终是阻断还是提醒。

## 4. 规则准入六问

新增或升级规则前回答：

1. 谁会消费它，行为会因此发生什么变化？
2. 来源是什么：负责人意图、真实事故、合同/合规，还是仅仅担忧？
3. 条件能否被机器可靠判定？不能就不要伪装成硬门禁。
4. 离风险最近的载体是什么？能用权限解决就不要只写 CI。
5. 谁能绕过，绕过是否必须留痕？
6. 何时删除、降级或复审？

安全、合规、不可逆操作和外部合同不需要等待“第二次使用”才沉淀；沉淀时机按因果机制普适性判断（参见 governance/cases 判例「机器节拍与数字门槛不继承人类惯性」）：机制普适的单例即可收编；机制存疑的先记判例观察，复用时自然显形。

## 5. 确定性与 AI 判断分离

### 确定性硬门禁

适合脚本、测试、权限和 CI：

- 文件是否存在、链接是否失效；
- 测试是否通过；
- API 是否返回禁出字段；
- 命令是否匹配危险前缀；
- 状态迁移是否满足有限状态机。

### AI 建议层

适合只读审计和人工抽检：

- 架构是否偏离意图；
- 新抽象是否值得；
- 判例能否回答新问题；
- 文档是否存在语义冲突。

AI 审计默认只读，输出固定结构并附证据。除非另有独立确定性判据，不把 LLM 单次结论作为唯一合并阻断条件。

## 6. 事故回路

事故不自动等于“新增一条规则”。先判断：

1. 既有载体是否根本没有触发；
2. 判定条件是否模糊或错误；
3. 覆盖范围是否不足；
4. 是否存在被允许的绕过；
5. 该风险是否值得继续接受。

同类问题第二次发生时，强制审计上一次机制；不机械地升级层级。修复可以是升级载体、修正判据、收窄范围、删除无效规则，或显式接受风险。

## 7. 单一真相与生成物

同一规范正文只有一个权威来源；其它位置使用链接或生成物。生成副本是允许的，但必须满足：

- 清楚标记生成来源；
- 可重复生成；
- CI 能检测漂移；
- 人不直接编辑生成物。

因此 Codex 与 Claude Code 可以有不同发行包，但不能维护两套手写核心方法论。

## 8. 人仍然负责什么

- 定义项目意图、品味和不可替代的取舍；
- 批准架构级、不可逆和高风险变更；
- 抽检 AI 的语义判断；
- 决定治理成本是否仍然值得。

机器负责明确触发和确定性阻断；AI负责准备材料和建议；人负责不能外包的意图与责任。

## 9. 结构治理与领域治理

治理不应吞掉软件架构和设计系统，但必须保证它们有单一真相、明确变更门和自动验证。

### 仓库结构

- `docs/architecture/repository-layout.md`说明顶层目录、所有者、允许内容和禁止边界；
- 新增顶层目录、新运行时或跨层依赖属于架构变化，进入ADR；
- 目录摆放规则能机器判断时进入lint，不能判断的只做review提示；
- 大型子树可放更近的`AGENTS.md`/运行时覆盖，避免根指令膨胀；
- 生成物、缓存、临时文件必须有固定位置并被ignore，不能散落项目根。

### 前端设计语言

设计系统是产品架构，不是治理正文。治理层只约束：

- 设计token、组件、交互和内容语言的权威位置；
- 新页面优先复用token和组件，新原语需要设计决策；
- token从代码或结构化数据生成多端产物，禁止手工维护多个副本；
- 可访问性、响应式、视觉回归和关键交互测试进入自动验证；
- 项目没有前端时不安装该扩展。

目录结构是公共底座；前端设计系统以可选extension安装，避免把领域规则强加给所有项目。
